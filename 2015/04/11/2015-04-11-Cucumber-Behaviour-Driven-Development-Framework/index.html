Cucumber: Behaviour Driven Development Framework

2015-04-11
College Relics

Cucumber is a software tool that allows the developer to write features for an application in a natural language. It is based on Behaviour Driven Development (BDD). Cucumber is available for the JVM and Python languages (among others) and was originally developed for Ruby on Rails.

Briefly, BDD is extended from the limitations of Test Driven Development (TDD). TDD describes a general process for writing a test followed by application code. However,it doesn’t specify what should be tested nor how tests should be structured. Furthermore, while TDD facilitates end user involvement in application development, it often can’t bridge the disconnect between the non-technical user viewpoint and the end product functionality. BDD addresses this disconnect by bridging the language barrier between business stakeholders and their technological counterparts.

Dan North coined the BDD concept and created the first BDD framework named ‘JBehave’ around 2008. The framework, as well as subsequent frameworks including Cucumber, specifies behaviour in terms of ‘user stories’ which include a narrative and acceptance criteria. Based on North’s work, the Behaviour Driven website (behaviourdriven.org) presents “a framework of activity based on three core principles:

"1. Business and Technology should refer to the same system in the same way - ItsAllBehaviour
 2. Any system should have an identified, verifiable value to the business - WheresTheBusinessValue
 3. Up-front analysis, design and planning all have a diminishing return - EnoughIsEnough"

Cucumber relies on a small vocabulary to get all viewpoints on the same page using the same words. The ‘cukes.info’ website epitomizes the process in 6 steps which are:

1. "Describe behaviour in plain text"
2. "Write a step definition..." (in Ruby)
3. "Run and watch it fail"
4. "Write code to make the step pass"
5. "Run again and see the step pass"
6. "Repeat 2-5 until green like a cuke"  ( which is when the code font color all turns to green).

Technically, this is accomplished by first creating a feature file (.feature). Within the feature, scenarios are created as a sequence of steps to describe what the program should do. There are three kinds of steps: ‘given’ steps, ‘when’ steps, and ‘then’ steps. The steps gradually transform the scenario into code.

An example BDD format, similar to Cucumber is borrowed from wikipedia and looks like this:

"Story: Returns go to stock

    In order to keep track of stock
    As a store owner
    I want to add items back to stock when they're returned

    Scenario 1: Refunded items should be returned to stock
    Given a customer previously bought a black sweater from me
    And I currently have three black sweaters left in stock
    When he returns the sweater for a refund
    Then I should have four black sweaters in stock

    Scenario 2: Replaced items should be returned to stock
    Given that a customer buys a blue garment
    And I have two blue garments in stock
    And three black garments in stock.
    When he returns the garment for a replacement in black,
    Then I should have three blue garments in stock
    And two black garments in stock"

The potential usefulness appears high. Of course the main benefit is to reduce the conflicts between the business and technical viewpoints. In regard to the learning curve, one must already be well rehearsed in a programming language such as Python or Ruby on rails. The learning curve consists mainly of extending unit testing methods to include ‘feature’ files and ‘scenario’ steps. An experienced developer should be able to determine if it enhances his/her performance in short order (days probably).
