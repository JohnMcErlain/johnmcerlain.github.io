Some Useful Models for the Programmer

2015-02-20
College Relics

Programmer Viewpoint

Models are developed to foster communication between stakeholders of all levels. The MHC-PMS models with corresponding diagram representations would have been beneficial to the requirements elicitation phase. They help answer the question, “Who needs to understand the system and at what level?”

Because I am approaching Software Engineering with a programmer/system analyst viewpoint, I would need to formulate conceptual models and use diagrams that would help me better understand the management of data in an object oriented system, such as the MHC-PMS.

Sommerville mentions a survey which states that 5 of the 13 (or so) UML diagrams “can represent the essentials” of a system, each by representing one of the 4 models,or perspectives of that system. While exploring the 5 main diagrams (activity’, ‘use case’, ‘sequence’, ‘class’, and ‘state’) I am attempting to understand how each might support the ‘programmer viewpoint’ versus a more abstracted non-technical viewpoint.

Perhaps designing software could start with the Highest level of abstraction (lowest level of granularity) and work downward (to lower abstraction and higher granularity). The potential process may follow: 1. Understanding the context using a simple contextual diagram; 2. Coding processes using activity diagrams; 3.Translating interactions into code using sequence and use case diagrams; 4. Documenting software components and developing design patterns using class diagrams; 5. Debugging system behavior using state diagrams.

THE CONTEXT. This model may be the simplest, yet it encompasses the entire network of systems. It shows the integration of the new system with existing systems. The MHC-PMS is connected to 6 other systems. Perhaps the most important detail for designers/programmers (and one I would include in such a model) would be the location of the various systems. This would elucidate network requirements and perhaps eliminate many questions during requirements elicitation. It would provide designers an early indication of the magnitude of the project and the general scope of how much hardware would be needed for the software and the platforms to be utilized. Fopr example, one enterprise database versus 7 distributed databases will require radically different designs.

The ACTIVITY DIAGRAM defines processes and therefore reveals object functionality to the programmer. It may be beneficial to include prospective classes itemized with the activity. For instance, the process of ‘involuntary detention’ includes many sub activities (inform patient rights, record detention decision, transfer to hospital, admissions etc), almost all of which will require data entry. ‘Form’ objects would be required. Why not list these object classes and their methods with the activity? By doing so, a cross reference to the class diagram (below) might aid understanding.

THE SEQUENCE DIAGRAM. This diagram represents the order of method calls. For the programmer, a sequence diagram could be made for every interaction in the system. To do so would “blow up” the dependencies of the classes to a highly detailed view and provide the steps necessary to provide method call implementations.

THE CLASS DIAGRAM shows the entities in the MHC-PMS system (patients, GP’s, hospital staff etc)and therefore the ‘potential’ of any object that may be created in the system. However, it only shows one (albeit highly detailed) view which would be like the picture of the building on the first page of a thick set of architectural plans. The ‘associations’ and ‘interactions’ shown are static views. It doesn’t show ‘state’, nor does it show where all these classes came from (how did they come to be defined?). Perhaps because the MHC-PMS is a data-centric application, the database could be constructed first, and the programming class definitions derived from it.

THE STATE DIAGRAM State helps conceptualize the result of events and transitions on the whole system. Even a simple program may have hundreds or thousands (or many more) events, each triggering many more method calls, and each of those calls in turn influencing the data of one or more objects; in a complex system, debugging can become overwhelming. The MHC-PMS may not be so complex to require state diagrams because the principal objects (patients) generally do not interact with each other in the system. For more complicated systems, State would be an extremely helpful diagram if it could be manifested dynamically, so that “stepping” through the method calls with their corresponding state changes could be color coded in real time with an adjustable timer (to slow down program execution).
