Efficient Book Search

2014-12-13
College Relics

“In this Module you are reading about the String and StringBuilder Classes.

Lets apply some of the concepts to a theoretical project.

You are assigned to create a search application for xyz publishers. The task is to be able to search on any string of words. The search base will be from over 1000 books contained in the publisher’s on-line library.

Based on what you’ve learned how would you begin to build a search application? What additional information would you need to know?

What algorithms would you consider for use? Why? “

As with anything in the world of software, I would neither attempt to “reinvent the wheel”, nor necessarily use non customized off-the-shelf algorithms.

I am beginning to think that all solutions to unique problems lie somewhere between the two.

As such, for the given problem of searching a proprietary publishers book database, I would ask, who has already published effective search algorithms for strings in large content repositories?

The first thought that comes to mind is Google. As I recall the story, Larry Page woke up one night with an incredible epiphany: you don’t have to search “everything” but rather reference the importance of a search target, and then search a smaller set of items ranked on importance. His algorithm became known as “pagerank”, of course named after his own name.

Pagerank uses a directed graph (sound familiar?). Pages are vertices of the graph. One page referencing another page is a directed edge. The algorithm counts the number of “links” pointing at one page or another and then ranks the “importance” of a page by that link count.

This concept could be extrapolated to our publisher’s database. General book content could be ranked by the number of other books that include specific key words/key phrases in “book descriptions” creating a hierarchy of content. The hierarchy is based on “bookrank” which would rank its imporatnce for key topics reference by the other books.

Once the search has been narrowed down to several books, then a more string intensive algorithm, such as KnuthtMorrisPrattStringmatcher might be used to find specific search phrases.

This would just be the beginning of the design process to create such a search program. Other information to know would be the publisher’s intent, such as how large they expect the book repository to grow to.

If I had to derive a string search algorithm, I would either employ Team A to develop, or “borrow” one from Google. After all, wouldn’t Google be the obvious second place (behind Team A of course) to seek string searching algorithms?

So, how does Google Search? We all have heard that Google uses a pageRank to Rank Pages. But here are a few more facts about Google searches:

  1. Pagerank is named after Larry Page, one of the two founders of Google.
  2. Pagerank does not use StringBuilder or String type classes nor the Knuth Pratt
  3. Pagerank uses graph representation in the pagerank algorithm.

Furthermore, these collections could be converted into key/value pairings.
