Looping Through Collections

2014-11-19
College Relics

“Now that we have learned a few things about Data Structures and Algorithms lets try the Duke exercise again. Please discuss how you would solve the problem.

What Data structures would work best? Why?

What Algorithms? Why?

What are some of the considerations that should be taken into account? Tradeoffs between one approach compared to another?”

Nested loops then, HashSets now. Clearly, nesting loops are not the way to go, leading only to quadratic time!

The data (and content files) are most likely stored in a massive data base owned and maintained by a content provider. I am guessing that if we were to rip the hood off a standard database, the source code of the database management system would be comprised of Hashsets and Hashmap data structures utilizing Set algorithms for data manipulation.

Our question of “how to search for John Wayne” becomes two separate lines of questioning:

1) how does the content provider organize all of the content (Hashtables? Hashmaps?

2) how does a standard data manipulation language (such as SQL) perform search and sort routines.

Here is a hypothetical scenario: the content provider maintains an extremely large database of thousands of tables including one massive table “Shows”. The primary key would be the title of any show in their content universe. Show attributes would include necessary information for any network that wanted to air a particular show. This would include Show Title, Genre (talkshow, movie, sitcom), Show Type (action, adventure, comedy etc), and Duration ( 15, 30, 60, 90, 120 min, 120+). The programming manager of a particular network targets a particular time of year and time of day when a certain type audience may be watching and determines it is right for a John Wayne western movie.

Yes, the search turns up John Wayne in Θ(1) time but it also turns up several hundred John Wayne “shows” including guest show appearances, interviews etc. What the search has really turned up is the content providers table of “John Wayne”,in which each show he has ever been in is the primary key.

There may be a hashcode for each movie, which works well if you already know the movie to select. If the program manager wanted to narrow the search, I would guess that the search engine would implement the set interface (of Java or C# or any other collections framework language). The Set interface specifies algorithms which appear to perform fundamental database operations such as UNION (s1.addAll() ), INTERSECTION (s1.retainAll() ), and SUBSET (s1.containsAll() ). To narrow down the search from ALL John Wayne shows, a series of set modifications would be made using the intersection method s1.containsAll(). Thus, John Wayne plus Western returns a smaller list… then,that set plus 120 min time duration yields yet a smaller set and so on. Each intersection operation would be done in Θ(log n) time due to the fact that the attributes are grouped in “blocks”, eliminating the need to iterate through each item.

Recapping this possible scenario: Hashmaps are the structure for primary/composite keys of a database table. Linked lists or Arraylists are the structures for table attributes. Finally, a hashset or some such structure implementing the set interface would be the structure used to manipulate the data (e.g. SQL).

The content provider’s database includes a table for each actor with attributes such as “movies”, “talk shows”, “interviews”, “TV shows” etc. Furthermore, lets assume it has a massive table, “Genre”, which will have “actor” (all actors) as a foreign key, “show type”, “show format”, “duration”, “description” (among others). A TV station (which contracts with the content provider) may fill its slots by using an algorithm which selects first by duration (incrementing 30 minute from 30 to 240+), then sorted by “genre”, then “type”.

Once we “find” John Wayne, do we stop? Are we satisfied watching the first return? If we keep iterating through the table index, we will obtain a new list with hundreds of John Wayne movies, guest appearances and TV shows. Again, the question for network executives is to fill a time slot with “some western”. So the list is filtered, taking the collection and searching only for “western”. This would be a Set method finding the IntersectionOf(John Wayne, western). This process would be executed one more time to search for the proper time slot. The final list may have 50 John Wayne movies which meet the criteria, so the final method randomly chooses one. Each of these operations are performed in Θ(1) time.

The Iterator() method is just that, a method implemented by many different classes, all of which implement the Iterable interface. However, the method is generic in the sense that it produces an iterator, which returns the object (collection) in some revised way.

Tables by duration: 30 min; 60 min; 90 min; 120 min; by Genre

Lets say, the algorithm randomly chooses the Genre “Western”, “two hour”, “action”. To do this….Hash Table

The list that is returned is still massive and would include a small percentage of John Wayne movies all suitable to fill the time slot. This time a different algorithm is employed to search the ArrayList. Again, this is a list of lists (records) but this time it is all John Wayne. The algorithm could be a quite simple random selection based on number of records and done in Θ(1) time.

Of course, the “heavy lifting” in this scenario has already been done by the DBMS implementing tables as HashSet and providing a nice table of Duke content, from which program TV can randomly choose, or print out listings from which we can choose.

In our exercise, we are simply searching for the Duke, so we are not directly concerned with sorting algorithms (though the search algorithm may depend on sorted data). Our algorithm must know the database schema, which is the relations among the tables (and there may be thousands of tables).
