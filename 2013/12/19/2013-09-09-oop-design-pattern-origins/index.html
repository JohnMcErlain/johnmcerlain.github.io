Paper: OOP Design Pattern Origins

2013-12-19
College Relics

2013-12-19-Paper-Design-Patterns.md
Design patterns are recognizable class structures used in programming, that when used in appropriate groupings, help programmers resolve computing problems; Recognizing and understanding design patterns help programmers cope with abstraction at the extreme level. They give programmers a vocabulary of phrases to describe the complex relationships between classes. The term “design patterns” was coined by four men, Erich Gamma, Richard Helm, Ralph Johnson and John Vlissides also known as the “The Gang of Four “, or “Go4”. Their definition of design patterns was written in their collaborative book, “Design Patterns”: “Design patterns represent solutions to problems that arise when developing software within a particular context” (1). Our textbook, Objects First With BlueJ, defines a design pattern as “a description of a common computing problem and a description of a small set of classes and their interaction structure that helps to solve the problem”(2) . As one gains experience in programming, certain techniques are developed and certain programming constructs reappear as solutions to new problems.
It should be noted that when the now famous book of The Gang of Four (Gof4) was published in 1995, Java was in its infancy. The book applies itself to Object Oriented Design (as opposed to programming) and often references C++ and Smalltalk. Remarkably, now almost 20 years since the original print, the concept of design patterns are still not only valid, but still widely used; and they are just as applicable to the Java language as they are to C++. In fact, in the evolution of the Java language, some design pattern methodology has been incorporated into the Java language structure such as the concepts of ‘implementation’ and ‘interface’. Since 1995, the book has been reprinted over 40 times with little modification.
Before design patterns are considered for a particular program, the classes for the program must be determined. According to our text book, “deciding what classes to use to implement a solution to a problem can be one of the most difficult tasks”. In regard to “inexperienced programmers” it also states, “Doing the initial design is seen as… annoying and people cannot wait to get it over with so that the “real work” can start. This is a very misguided picture”. It goes on to show methodology using ‘Class/Responsibilities/Collaborators (CRC) cards to illuminate the interactions between classes. The ‘class’, of course, specifies the object. The ‘responsibilities’ are the things that object is accountable for. The ‘collaborators’ are the other classes that the object references. Though system design is beyond the scope of this course and project, it is mentioned as a necessary prelude to incorporating design patterns into a program. Class and program design is the foundational layer upon which the application is created. Our course helps us formulate the concept of CRC by grouping classes with United Modeling Language (UML). This is a good way of thinking about design patterns. A design pattern is a grouping of classes that can do a variety of things. By understanding these groupings, programmers can communicate more effectively with each other when designing programs.
The Go4 based the concept of design patterns on the writings of an architect, Christopher Alexander. Alexander is quoted as saying “Each pattern describes a problem which occurs over and over again in our environment, and then describes the core of the solution to that problem, in such a way that you can use this solution a million times over, without ever doing it the same way twice”. He was referring to buildings and towns. The Go4 contends the same applies to object oriented design, except that “Our solutions are expressed in terms of objects and interfaces instead of walls and doors”(1). “At the core of both kinds of patterns is a solution to a problem in a context”(1).
Understanding Design Patterns facilitate our dealings with ‘abstraction’. The authors have identified twenty-three unique, but sometimes interrelated design patterns. All of them are identified by purpose and scope. As stated, the main purpose in programming is to solve computing problems. However, the purpose of a particular design pattern, is abstracted from this process. There is never a design pattern that can be copied from one application and plugged into another to resolve an issue. In this way, design patterns can be regarded within the concept of ‘coupling’; intangible and subjective. A design pattern should be viewed more as a tool that may be applied in a certain circumstance and may (or may not) help resolve an issue. In this regard it is similar to “toolkits” which are reusable classes, found in various class libraries, which provide general purpose functionality. However, the “purpose” of a design pattern is discussed in the context of designing objects and classes to work together toward a specific objective.
All twenty three patterns are categorized under only one of three main purposes: Creational, Structural, or Behavioral. Within each of these three categories, they are further divided into “scope”; that is, whether the pattern focuses more on class or object creation, object structure or object behavior; or a focus on the instance of the class being the object’s creation, object’s structure or object’s behavior. Nineteen of the twenty-three patterns focus on the object.
Briefly, the three main categories are explained as follows:
Creational: these are mechanisms, which attempt to create objects tailored to a given situation. Introducing new objects into a system can pose risk to the integrity of that system. Creational design patterns control object creation and therefore the risk.
Includes: Factory method, Builder; Prototype; and Singleton;

Structural: these patterns define relationships among objects and classes; they build and develop the hierarchical inheritance patterns; they identify simple ways to realize relationships between entities.
Includes: Adapter; Bridge; Composite; Decorator; Facade; Flyweight; Proxy;

Behavioral: these design patterns identify common communication patterns between objects and realize by doing so, they increase flexibility in carrying out this communication.
Include: Interpreter; Chain of responsibility; Command; Interpreter; Iterator; Mediator; Memento; Observer; State; Strategy; Template method; Visitor.

Before launching into a discussion of the categories and how design patterns solve design problems, it is important to understand the complexity of object oriented programming.  The data structures derived from (even) simple class structures can become complicated fast.  A compiler builds a graph to represent relationships between classes and objects; connections are represented by the possibility of a method of one class calling a method from another class(5).  The graph would be quite simple if the program consisted of just a few classes such as that of a parking meter program, a music player system or a bouncing ball program.  But what happens when the program is very large consisting of hundreds, or thousands of inter-related objects, many influencing each other?  The compiler graph becomes very complex.  It references object data.  It also stores the relationships among methods that operate upon the data.  If code is written whereby one class calls another, the compiler will substantiate this only if the graphical representation supports it.  
  The dilemma for programmers becomes one of dealing with the complexity as they try to fathom a solution for their application.  As we have learned, the issue of abstraction, itself quite complex, aids programmers on several levels.  First of all, it breaks down large complex tasks into smaller, more manageable tasks.  Fundamentally, design patterns do the same.  They provide names for certain class and object interactions so we as programmers can discuss projects with one another.
Design patterns solve design problems.  This is done by determining what objects to use, determining object implementation and/or object interfaces, and therefore developing reusable code that also accounts for future change in program functionality.  These objectives may appear overwhelmingly complex, and the Gof4 defines two principles to aid the process.  The first of these involves inheritance and states "program to the interface, not (to) an implementation".  The second of the main principles involves object composition and states "Favor object composition over class inheritance".
  To understand the first of these principles, one must understand the concept of "interface" which itself has rapidly evolved over the past decade with multiple meanings.  The principle refers to the original, most fundamental meaning, "The set of all signatures defined by an object's operations (methods).  The interface describes the set of requests to which an object can respond".  This is nothing more than hierarchizing classes, either by means of inheritance or by means of utilizing a Java 'interface', or by means of utilizing  the Java "implements" keyword implementing a unique class that brings unrelated classes together into one interface.  "Programming to an implementation can cause problems and more work when the class is redesigned or removed leaving the implementation "dangling" causing an error message."  Therefore, when using inheritance, it is generally good methodology to "program to the interface".
Here is an example of programming to an interface versus to an implementation:

public static void main(String… args) {
// do this - declare the variable to be of type Set, which is an interface
Set buddies = new HashSet();
// don’t do this - you declare the variable to have a fixed type
HashSet buddies2 = new HashSet();
} (6).
As can be seen from above, if the collection had to be changed from a HashSet to a LinkeedHashSet, then not only would the variable, buddies2 have to be changed to LinkedHashSet buddies2 = new LinkedHashSet(), but all the return types of all the methods of the “Buddy” class. “Reuse by inheritance makes it easier to make new components that can be composed with older ones”.
Inheritance works hand-in-hand with object composition and therefore the second principle, “Favor object composition over class inheritance”. In so doing, the program will have fewer classes and more objects. The system’s behavior will depend more on object interaction instead of class definitions. This underlying concept of design patterns also facilitates “reusability” of the code and designing for change.
In fact, it is a central theme for experienced programmer’s to have the mindset to design a program that is flexible, and can conform and evolve to future changes. This is where specific design patterns can be named to address specific design issues. For instance, in the example of the LinkedHashSet above, three specific design patterns help the programmer NOT specify a class explicitly. They are “Abstract Factory”, Factory Method” and “Prototype”. To help avoid situations of hard coding requests, a programmer would consider “Chain of Responsibility” and “Command”. To avoid cases of tight coupling, the programmer would consider “Abstract Factory”, “Bridge”, “Chain of Responsibility”, “Command”, “Facade”, “Mediator”, and/or “Observer”. In cases where algorithms are used and likely to be modified, patterns to consider would be “Builder”, “Iterator”, “Strategy”, “Template Method”, and/or “Visitor”. When one is using a class library and cannot access the source code, design patterns including “Adapter”, “Decorator”, and “Visitor” offer ways to modify classes.
There are guidelines to use when selecting a pattern. At the very minimum when considering a design pattern to integrate into your project, one must consider the name of the pattern, the problem description, the solution description and the consequences of using one pattern versus another pattern(2). For more detailed information, the catalog of design patterns will provide additional information such as the applicability of the pattern, the UML structure (somewhat similar to BlueJ), the participants, collaborations, consequences, implementation, sample code, known uses and related patterns.
As ubiquitous as the concept has become, the concept has its detractors. During the Object-Oriented Programming, Systems, Languages & Applications (OOPSLA) Conference of 1999, the Go4 was charged with “numerous crimes against computer science” and “convicted” by 2/3s of the “jurors” who attended the “trial”, though it is unclear what the sentence was. Clearly, coders have a large pattern of humor. The verdict: Patterns are through the “hype cycle”. The GOF design patterns are no longer blindly touted as useful everywhere, but instead are seen as part of the designer’s toolbox, to be understood and used where appropriate. Peter Norvig provides a similar argument. He demonstrates that 16 out of the 23 patterns in the Design Patterns book (which is primarily focused on C++) are simplified or eliminated (via direct language support) in Lisp or Dylan. (4) It should also be noted that since 1995, and the writing of “Design Patterns”, the Java language has included “interface” as part of the language: A (Java) programming language keyword used to define a collection of method definitions and constant values. It can later be implemented by classes that define this interface with the “implements” keyword. Some of the most important Java interfaces are “Collection”, “Iterator”, “List”, “Map”, and “Set”.
My interest in this topic was to gain a foothold in Java as fast as possible. Since design patterns are the lingo used by programmers, I figured that learning some of the essential core lingo as a newbie Object oriented student might help get me up to speed faster as a pro active learner of the subject. One can only imagine what the programming language “Java”, or its successor will be like in 100 years. Massive servers would be needed to store an unfathomable number of classes and methods developed through-out the 21st century. Programmers will work at a level of abstraction that we can’t begin to understand with our “primitive” mindsets. If Object Oriented Programming is difficult to learn now, how will students crack coding in the future? The answer to this question is answered in part by the process of abstraction and use of design patterns.
